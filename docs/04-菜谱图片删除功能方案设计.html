菜谱图片删除功能方案设计

为已上传图片补充可清空、可保存、可审计的删除路径
Feature: Remove Recipe Image Version: Draft 1 Date: 2026-07-23
目录： 背景与问题 目标范围 方案比较 推荐方案 实施拆解 测试计划 风险与决策点

1背景与问题

当前编辑菜谱页面允许上传或替换图片，但没有提供“删除图片”的入口。用户一旦给菜谱绑定图片，只能换成另一张图片，不能恢复到无图状态。

现状证据

  • src/app/recipes/[id]/edit/page.tsx 中图片区域只有预览和文件上传控件，没有清空按钮。
  • 编辑页保存时传递 imageUrl: imageUrl || undefined，无法明确表达“删除图片”。
  • src/lib/validators.ts 当前会把 imageUrl: null 转换成 undefined；这对兼容读取有用，但不适合作为更新接口的删除语义。
  • 上传接口只支持 POST /api/upload，没有用于删除 public/uploads 文件的接口。
  • src/lib/actions/recipes.ts 当前权限查询只读取 userId 和 authorRole；物理文件清理需要后端同时读取更新前的 recipes.imageUrl 作为可信旧值。

2目标范围

核心目标
用户可在编辑菜谱时移除当前图片，并保存为无图菜谱。
数据目标
云端菜谱的 image_url 被置为 NULL；本地菜谱的 IndexedDB imageUrl 被清空。
权限目标
只有有权编辑该菜谱的用户才能删除图片。
文件目标
优先保证业务数据正确；物理文件删除采用保守策略。

不纳入本次范围

  • 不做图片库、历史图片恢复、图片批量清理后台。
  • 不改变菜谱图片上传格式、大小限制、压缩策略。
  • 不删除外部 URL 图片资源，因为应用不拥有外部资源。
  • 不保证清理新建页上传后又取消保存产生的孤儿文件；这是本次的明确取舍，可后续用定期清理任务解决。

3方案比较

方案 内容 优点 风险 结论
A. 仅前端清空引用 编辑页加“删除图片”按钮，保存时把 imageUrl 清空。 实现简单，用户立即可用。 需要调整后端更新语义；上传文件会残留。 不完整
B. 清空引用 + 保守文件清理 保存时把 image_url 置空；如果原图是本应用 /uploads/ 文件，并且没有其他菜谱引用，再尝试删除物理文件。 用户体验完整，减少无用文件，不会误删共享图片。 实现略多，需要文件路径安全校验和引用计数查询。 推荐
C. 独立图片删除 API 新增 DELETE /api/upload 或 DELETE /api/recipes/:id/image，点击按钮立即删除。 语义清晰，可独立重试。 编辑表单未保存时容易出现“图片已删但表单取消”的不一致；需要更多状态管理。 暂不推荐

4推荐方案

采用方案 B：清空引用 + 保守文件清理

用户在编辑页点击“删除图片”后，页面立即隐藏预览，并显示可撤销的无图状态；只有点击“保存修改”后，菜谱记录才真正变为无图。 后端更新接口需要明确支持 imageUrl: null，并把数据库字段更新为 NULL。

对物理文件删除采取“最佳努力”策略：后端必须先从数据库读取更新前的 oldImageUrl，如果旧图片地址是 /uploads/<filename>，且没有其他菜谱继续引用同一路径，后端在成功更新数据库后尝试删除对应文件。 文件删除失败不回滚菜谱保存，只记录服务端日志，避免因为磁盘清理失败影响用户完成编辑。

文件清理只针对 oldImageUrl，不能使用客户端传入值或更新后的值作为删除目标。只有当 oldImageUrl 与保存后的新值不同，且 oldImageUrl 通过严格路径校验时，才进入物理文件清理流程。

这样可以同时满足用户视角的“删除图片”和系统视角的数据安全，避免误删被官方菜谱、复制菜谱或导入菜谱复用的同一图片文件。

更新接口语义

PUT /api/recipes/:id

{
  "title": "番茄炒蛋",
  "imageUrl": null,
  "ingredients": [{ "name": "番茄", "amount": "2个" }],
  "steps": [{ "phase": "cook", "text": "炒熟" }],
  "tags": ["快手菜"]
}
输入 含义 数据库结果
imageUrl: "..." 设置或替换图片 image_url = "..."
imageUrl: null 删除图片引用 image_url = NULL
imageUrl 缺失 本次更新不改变图片字段 保持原值
客户端类型和保存封装需要同步表达该语义：UnifiedRecipe.imageUrl 可继续用于读取为 string | undefined，但 saveRecipe 的入参需要允许 imageUrl?: string | null。编辑页需要增加 imageRemoved 或等价状态；用户点击删除后，保存 payload 必须发送 imageUrl: null，不能继续使用 imageUrl || undefined。

服务端旧值读取与清理顺序

  1. 在 updateRecipe 的授权查询中同时选出 recipes.imageUrl，保存为 oldImageUrl。
  2. 在同一个事务内完成权限校验、菜谱字段更新、食材/步骤/标签更新。
  3. 事务成功后，根据请求语义计算保存后的 newImageUrl：字符串表示新图，null 表示无图，字段缺失表示沿用 oldImageUrl。
  4. 只有 oldImageUrl 存在且 oldImageUrl !== newImageUrl 时，才尝试对 oldImageUrl 做引用计数和物理文件清理。

5实施拆解

  1. 调整后端校验模型。 将创建和更新的图片字段语义拆开：创建时可以继续接受缺省或字符串；更新时保留 null，不要把 null 统一转成 undefined。
  2. 扩展客户端保存入参。 将 saveRecipe 可接受的图片字段扩展为 imageUrl?: string | null。读取模型仍可把无图表示为 undefined，但写入模型必须保留 null 作为删除命令。
  3. 调整 updateRecipe 入参和更新逻辑。 授权查询时选出 recipes.imageUrl 作为 oldImageUrl。当请求体包含 imageUrl 字段时，按字符串或 null 更新数据库；当请求体未包含该字段时，不修改图片字段。
  4. 增加上传文件清理工具函数。 函数只接受严格格式 /uploads/<filename>，其中 <filename> 必须是单段文件名，解码后不能包含 /、\、..，并且必须匹配当前上传格式 nanoid(16).jpg|png|webp。任何不符合条件的路径都直接拒绝，不做 path.basename 静默改写。
  5. 增加引用计数保护。 事务成功后只对 oldImageUrl 做清理。删除物理文件前查询是否还有其他菜谱引用相同 image_url。只有 oldImageUrl !== newImageUrl 且引用数为 0 时才尝试删除文件。
  6. 编辑页增加“删除图片”按钮。 仅在当前有图片时展示。点击后清空本地 imageUrl 状态，设置 imageRemoved 或等价状态，并展示“保存后删除图片”的提示；用户点取消返回时不提交变更。
  7. 保存 payload 明确发送 null。 编辑页需要区分“没有改图片”和“删除图片”。当用户点击删除后，保存请求发送 imageUrl: null。
  8. 补充本地菜谱行为。 对 local-* 菜谱，删除图片只清空 IndexedDB 记录中的 imageUrl，不做物理文件删除；本地模式下原本也不支持通过上传接口写入新图片。
  9. 保持新建页简单。 新建菜谱页可以同步增加“移除已选图片”按钮，用于撤销本次上传或选择；它不触发物理删除，保存时无图即可。若图片已上传但用户没有保存菜谱，会留下孤儿文件，这是本阶段接受的产品取舍。

6测试计划

单元测试

  • recipeUpdateSchema 解析 imageUrl: null 后仍保留 null。
  • recipeUpdateSchema 在缺失 imageUrl 时返回 undefined，用于“不改图片”。
  • 图片清理工具只接受 /uploads/<nanoid16>.jpg|png|webp 这类应用内路径，拒绝外部 URL、绝对路径、多段路径、反斜杠路径、../ 路径、解码后包含路径分隔符的值，以及不符合当前上传文件名格式的值。
  • 当其他菜谱仍引用同一图片时，不删除物理文件。
  • 当引用数为 0 且文件存在时，删除文件；文件不存在时不抛出用户可见错误。

API / Action 回归测试

  • PUT /api/recipes/:id 或 updateRecipe 接收 imageUrl: null 时，将当前用户有权编辑菜谱的 image_url 清空。
  • 请求体缺失 imageUrl 时，保留旧图片 URL，不误清空。
  • 替换图片时，数据库写入新 URL；事务成功后只对 oldImageUrl 做清理，不能清理新 URL。
  • 无权限用户提交 imageUrl: null 时返回权限错误，数据库图片字段保持不变，也不触发文件删除。
  • 本地 local-* 菜谱删除图片后，IndexedDB 记录不再包含 imageUrl，且不会调用云端物理文件清理。

手动验收

  1. 登录后创建一条带图片菜谱，进入详情页确认图片展示。
  2. 进入编辑页，点击“删除图片”，确认预览消失且保存按钮可用。
  3. 点击“保存修改”，回到详情页确认不再展示图片。
  4. 刷新页面后再次进入详情页，确认图片仍然为空。
  5. 用无权限用户访问其他人的菜谱编辑接口，确认不能删除图片。
  6. 同一图片被两个菜谱引用时，删除其中一个菜谱的图片，确认另一个菜谱仍能正常显示。
  7. 在访客模式创建本地菜谱并编辑删除图片，确认保存后本地详情页恢复为无图状态。
  8. 在新建页上传图片后点击“移除已选图片”再保存，确认菜谱为无图；该步骤不验收物理文件删除。

7风险与决策点

风险 处理方式
误删共享图片文件 事务成功后只对 oldImageUrl 清理；删除前查询其他菜谱引用；仍被引用则只清空当前菜谱字段，不删文件。
路径穿越或删除非上传目录文件 只接受严格格式 /uploads/<nanoid16>.jpg|png|webp；解码后包含 /、\、.. 或不符合上传文件名格式时直接拒绝，不做 path.basename 改写。
用户点击删除后又取消编辑 删除动作先作用于表单状态；只有保存后才提交后端变更。
物理文件删除失败 不回滚菜谱保存；记录服务端日志，后续可由清理任务处理。
外部图片无法删除 只清空引用；不尝试删除应用不拥有的外部资源。
本地菜谱没有云端文件所有权 本地菜谱只清空 IndexedDB 中的 imageUrl，不执行物理文件删除。
新建页上传后取消或移除图片产生孤儿文件 本阶段接受该取舍；新建页移除按钮只影响表单状态。后续若需要彻底治理，再设计定期清理未引用上传文件的任务。
待审查决策： 是否同意本次实现包含“物理文件保守清理”？如果希望先快速交付，可以第一阶段只清空菜谱图片引用，第二阶段再补充无引用文件清理。
