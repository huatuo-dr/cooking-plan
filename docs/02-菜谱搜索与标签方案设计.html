菜谱搜索与标签方案设计

为菜谱浏览和 Cooking 计划选择增加快速定位能力
版本：评审稿 日期：2026-07-22 范围：搜索、标签、筛选
设计结论 现状分析 功能范围 数据模型 交互设计 接口与数据流 迁移策略 测试计划 风险与决策

1设计结论

推荐方案：标签持久化保存，搜索和标签筛选在前端完成。云端菜谱使用规范化标签表，本地匿名菜谱使用 tags: string[] 保存。首页和创建 Cooking 计划页复用同一套过滤逻辑。
为什么不先做后端搜索？
当前菜谱规模适合前端过滤，复杂度更低，体验也足够快。
标签编辑核心要求
编辑菜谱时先展示已有标签，同时允许输入并创建新标签。

2现状分析

  • 首页 src/app/page.tsx 通过 getAllRecipes() 获取菜谱后直接展示卡片列表。
  • Cooking 计划创建页 src/app/cooking/new/page.tsx 也通过 getAllRecipes() 获取菜谱后展示选择列表。
  • 云端菜谱存在 recipes、ingredients、steps 三类数据，当前没有标签字段。
  • 匿名用户菜谱保存在 IndexedDB，当前 LocalRecipe 也没有标签字段。
  • 官方菜谱由管理员用户创建，前端根据作者角色标记为官方。

3功能范围

本期包含

  • 菜谱支持多个标签。
  • 新建和编辑菜谱时可以添加、删除标签。
  • 编辑标签区域先展示已有标签，用户可点选已有标签，也可输入新标签创建。
  • 首页支持关键词搜索和标签筛选。
  • 创建 Cooking 计划页支持关键词搜索和标签筛选。
  • 搜索匹配菜谱名、食材名、标签名。
  • 筛选状态下仍保留已选择的 Cooking 菜谱，避免用户筛选切换时丢失选择。

本期不包含

  • 全文搜索步骤内容。步骤文本较长，默认纳入搜索容易产生噪声。
  • 标签管理后台。标签通过菜谱编辑自然创建和复用。
  • 分页、排序规则重做、推荐算法。

4数据模型

云端数据库

新增两张表，避免把标签直接塞进 recipes 字符串字段，方便去重、展示已有标签和后续扩展。唯一性约束使用规范化后的 normalized_name，展示时仍使用 name。

recipe_tags
- id serial primary key
- name varchar(20) not null
- normalized_name varchar(20) not null unique
- created_at timestamp not null default now()

recipe_tag_relations
- recipe_id integer not null references recipes(id) on delete cascade
- tag_id integer not null references recipe_tags(id) on delete cascade
- primary key (recipe_id, tag_id)
- index (recipe_id)
- index (tag_id)

Drizzle schema 需要显式声明复合主键和索引，避免实现时只靠字段注释表达约束。

export const recipeTagRelations = pgTable('recipe_tag_relations', {
  recipeId: integer('recipe_id').notNull().references(() => recipes.id, { onDelete: 'cascade' }),
  tagId: integer('tag_id').notNull().references(() => recipeTags.id, { onDelete: 'cascade' }),
}, (table) => ({
  pk: primaryKey({ columns: [table.recipeId, table.tagId] }),
  recipeIdIdx: index('recipe_tag_relations_recipe_id_idx').on(table.recipeId),
  tagIdIdx: index('recipe_tag_relations_tag_id_idx').on(table.tagId),
}))

本地 IndexedDB

匿名菜谱直接在 LocalRecipe 上增加标签数组，历史本地数据读取时兼容为空数组。saveRecipe() 的本地保存路径必须把 recipe.tags 写入 LocalRecipe.tags，否则匿名新建或编辑菜谱会丢标签。

interface LocalRecipe {
  id?: number
  title: string
  imageUrl?: string
  ingredients: { name: string; amount?: string }[]
  steps: { phase: 'prep' | 'cook'; text: string }[]
  tags: string[]
  createdAt: string
  updatedAt: string
}

统一前端类型

interface UnifiedRecipe {
  id: string
  title: string
  imageUrl?: string
  ingredients: { name: string; amount?: string }[]
  steps: { phase: 'prep' | 'cook'; text: string }[]
  tags: string[]
  createdAt: string
  source: 'local' | 'cloud'
  isOfficial?: boolean
  isMine?: boolean
}

5交互设计

首页浏览菜谱

  • 在菜谱列表上方增加一行过滤工具。
  • 左侧为搜索框，placeholder 为“搜索菜名、食材或标签”。
  • 下方或右侧展示可用标签 chip，点击后加入筛选条件。
  • 已选标签显示为激活态，支持再次点击取消。
  • 多个标签默认使用 AND 逻辑：只展示同时包含所有已选标签的菜谱。
  • 搜索和标签筛选同时生效。
  • 无结果时显示“没有找到匹配菜谱”，并提供清空筛选按钮。

创建 Cooking 计划

  • 在“选择菜谱”区域顶部增加相同的搜索框和标签筛选。
  • 菜谱列表只展示过滤后的结果。
  • 已选数量基于全部已选菜谱计算，不受当前过滤条件影响。
  • 如果某道已选菜谱被筛选隐藏，选择状态仍保留，提交时仍会加入计划。

新建和编辑菜谱标签

  • 在菜谱图片和食材之间增加“标签”区域。
  • 先展示已有标签，来源必须是当前用户可见菜谱的标签集合。
  • 已有标签可以直接从 getAllRecipes() 返回的可见菜谱中聚合；如果后续增加标签建议接口，后端也必须复用菜谱可见性条件，不能直接返回全局 recipe_tags 表。
  • 点击已有标签即可添加到当前菜谱，再次点击或点击标签上的删除按钮可移除。
  • 输入框允许创建新标签，按 Enter、逗号或点击“添加”完成创建。
  • 新标签会立即显示为当前菜谱标签，保存菜谱后持久化。
  • 前端提交前调用共享 normalizeTags()，后端 validator 和写入逻辑也必须再次调用同一个规范化函数。
  • normalizeTags() 放在纯共享模块 src/lib/recipe-tags.ts，不能依赖 React、Next server APIs、数据库或浏览器 API，确保客户端表单、服务端 validator、服务端写入和过滤函数测试使用同一套规则。
  • 规范化规则：去首尾空格、把中文全角空格转为空格、合并连续空白、大小写折叠生成 normalizedName、去重、过滤空标签。

建议限制：单个标签最多 20 个字符，每个菜谱最多 8 个标签。这样可以避免卡片和选择列表被标签撑开。

详情页展示完整标签；首页卡片和 Cooking 选择列表只展示前 3 个标签，超出部分显示“+N”。这样用户编辑后能看到完整反馈，同时列表布局不会被撑开。

6接口与数据流

云端接口

接口变化说明
GET /api/recipes 返回 tags 每个菜谱返回标签数组，供首页和 Cooking 计划页过滤。
GET /api/recipes/:id 返回 tags 编辑页进入时展示当前菜谱已有标签。
POST /api/recipes 接收 tags 创建菜谱时同步创建或复用标签。src/lib/storage/client.ts 的 saveRecipe() 云端 payload 必须包含 tags: recipe.tags ?? []。
PUT /api/recipes/:id 接收 tags 更新菜谱时重建该菜谱的标签关联。saveRecipe() 更新路径也必须传递 tags。
POST /api/recipes/copy 复制 tags 复制官方菜谱或可见菜谱时，除 title、image、ingredients、steps 外，也要把原菜谱标签传给 createRecipe()。

后端写入规则

  • recipeSchema 增加 tags 字段校验，并通过共享 normalizeTags() 转换，防止绕过 UI 写入脏标签。
  • createRecipe() 和 updateRecipe() 在同一个数据库事务内写菜谱、食材、步骤和标签关联。
  • 创建或复用标签必须使用事务内 upsert：按 normalized_name 执行 on conflict do nothing，随后按规范化名称批量查询标签 ID。
  • 更新菜谱时先删除该菜谱旧的 recipe_tag_relations，再插入新关联；不删除未被引用的标签记录，避免并发和历史引用问题。
  • GET /api/recipes 已经批量查询食材和步骤，标签也必须按当前可见菜谱 ID 一次性批量查询并分组，不能在 allRecipes.map() 中逐条查询。

前端共享过滤逻辑

新增纯函数模块，首页和 Cooking 计划页共同使用，避免两处实现不一致。

filterRecipes(recipes, {
  query: string,
  selectedTags: string[]
})

匹配规则：
1. query 为空时不限制关键词。
2. query 命中菜谱名、食材名或标签名即匹配。
3. selectedTags 为空时不限制标签。
4. selectedTags 非空时，菜谱必须包含所有已选标签。

客户端保存路径

标签不能只停留在页面组件状态里，所有创建、编辑、复制路径都要显式经过统一类型。

const payload = {
  title: recipe.title,
  imageUrl: recipe.imageUrl,
  ingredients: recipe.ingredients || [],
  steps: recipe.steps || [],
  tags: normalizeTags(recipe.tags || []).map(tag => tag.name),
}

const localRecipe: LocalRecipe = {
  ...
  tags: normalizeTags(recipe.tags || []).map(tag => tag.name),
}

7迁移策略

  1. 数据库 schema 增加标签表和关联表 使用 Drizzle schema 描述新表。当前项目开发和生产入口都采用 Drizzle push 流程，开发运行 npm run db:push，生产容器启动时由 scripts/entrypoint.sh 自动执行 schema 同步；如果后续切换为 migration 流程，再补充正式迁移文件。
  2. 兼容旧菜谱 现有云端和本地菜谱默认标签为空数组，不影响现有访问。
  3. 官方菜谱标签 第一版不强制自动补标签，管理员可以进入编辑页手动添加。若评审通过，也可以为已导入的 36 道官方菜谱补一份初始标签脚本。
  4. 本地 IndexedDB 升级 Dexie 版本递增，读取旧数据时用 recipe.tags ?? [] 兜底。

8测试计划

测试对象重点验证
标签规范化 去空格、全角空格处理、连续空白合并、大小写折叠、去重、过滤空标签、20 字符长度限制、每菜最多 8 个标签。
过滤函数 关键词匹配菜名、食材、标签；多标签 AND 筛选；无条件返回全部。
云端保存 创建菜谱保存标签；编辑菜谱替换标签；复制菜谱保留标签；删除菜谱级联删除关联；并发创建相同标签不会 500。
本地保存 匿名菜谱保存标签；旧本地菜谱没有标签时正常展示。
Cooking 选择页 筛选后已选菜谱不丢失，提交计划使用全部已选项。
可见性与性能 已有标签只来自可见菜谱；不返回其他用户私有标签；菜谱列表标签查询无 N+1。

9风险与待确认点

主要风险

  • 如果标签不规范，后续会出现“牛肉”“牛 肉”“牛肉类”等重复语义标签。第一版通过共享 normalizeTags()、normalized_name 唯一约束和已有标签点选降低这个问题。
  • 全局 recipe_tags 表意味着同一个 normalized_name 共享一个展示名。例如先创建 beef 后创建 Beef 时，展示名可能沿用已有记录。当前接受这个取舍，因为标签主要用于筛选和复用；如后续需要保留每个菜谱提交时的展示文本，再把展示名下放到关联表或改成按用户隔离标签。
  • 标签太多会影响卡片布局，因此每道菜最多 8 个标签，列表只展示前 3 个标签，详情页展示完整标签。
  • 前端过滤适合当前规模。如果未来菜谱数量增长到上千级，再迁移到后端搜索和分页。

待确认点

  • 官方 36 道菜是否需要我在实现时顺手整理一版初始标签。
  • 多标签筛选是否接受默认 AND 逻辑。如果你更希望“任意一个标签命中即可”，可以改为 OR。
  • 搜索是否需要包含步骤文本。当前建议不包含，以免结果过宽。
