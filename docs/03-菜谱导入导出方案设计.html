菜谱导入导出方案设计

用通用 JSON 菜谱包实现跨环境迁移、备份和批量分享
版本：评审稿 日期：2026-07-22 范围：菜谱 JSON 导入/导出
设计结论 目标与边界 JSON 格式 导出设计 导入设计 校验策略 接口与模块 迁移使用 测试计划 风险与决策

1设计结论

推荐方案：实现一个通用“菜谱包 JSON”导入/导出能力。JSON 只描述菜谱内容本身，不保存 id、userId、source、isOfficial、isMine 等账号和环境字段。导入后归属由当前用户身份和操作入口决定。
数据载体
版本化 JSON 文件，后缀建议 .recipes.json
导出范围
基于当前可见菜谱选择，支持全选和手动多选
导入归属
匿名导入本地；普通用户导入“我的菜谱”；管理员导入官方菜谱
冲突策略
不覆盖同名菜谱，自动追加 _2、_3 后缀

2目标与边界

必须满足

  • 菜谱内容导出时不区分“官方、本地、我的”，这些归属信息不进入 JSON。
  • 导出时可以选择任意数量菜谱，也可以一键全选当前列表中的菜谱。
  • 导出和导入均使用 JSON，格式稳定且带版本号。
  • 导入时必须先校验 JSON 文件合法性，校验通过后才允许写入。
  • 标签随菜谱一起导入/导出，继续复用现有标签规范化规则。
  • 管理员导入固定写入官方菜谱，因为当前系统通过作者角色 authorRole === 'admin' 推导官方身份，没有“管理员个人菜谱”的数据模型。

不纳入第一版

  • 不导出用户、密码、邀请码、Cooking 计划、Cooking 会话或任何账号数据。
  • 不打包图片二进制文件。imageUrl 只作为字符串引用导出。
  • 不支持 CSV、Excel、HTML 等格式。
  • 不做跨版本复杂转换，只支持当前版本和必要的向后兼容。
  • 第一版不支持覆盖同名菜谱，也不提供“一键清空并导入官方菜谱”。

说明：本功能可以覆盖“本地满意的官方菜谱迁移到线上”的场景，但实现方式是内容级迁移，不是数据库备份。

3JSON 格式

定义顶层 RecipeExportPackage，通过 version 标记格式版本。第一版只包含菜谱内容和最少元数据。

{
  "version": 1,
  "exportedAt": "2026-07-22T07:00:00.000Z",
  "app": "cooking-plan",
  "recipes": [
    {
      "title": "紫菜炒饭",
      "imageUrl": null,
      "ingredients": [
        { "name": "米饭", "amount": null },
        { "name": "紫菜", "amount": null }
      ],
      "steps": [
        { "phase": "cook", "text": "炒米饭，蚝油、生抽" }
      ],
      "tags": ["超级单品"]
    }
  ]
}
字段规则说明
version必须为数字，第一版为 1后续格式变化时用于兼容判断。
exportedAtISO 字符串仅用于展示和排查，不参与业务判断。
app可选字符串，建议为 cooking-plan导入时仅用于提示文件来源，不作为准入条件；非 cooking-plan 来源展示确认提示。
recipes非空数组，最多 100 项每项为一份菜谱内容，客户端和服务端都必须执行相同上限。
title1-200 字符导入后作为菜谱名称。
imageUrl字符串或 null第一版只保存引用，不复制文件。
ingredients1-100 项name 必填，amount 可为空。
steps1-100 项phase 只允许 prep 或 cook。
tags0-8 项复用 normalizeRecipeTags()，标签长度和规范化长度均不超过 20。

4导出设计

入口

在首页菜谱列表增加“批量导出”模式。进入后，当前列表卡片显示复选框，工具栏展示已选数量、全选、取消全选、导出 JSON、退出批量模式。

选择范围

  • 默认对当前用户可见菜谱操作：匿名用户为本地菜谱 + 官方菜谱，登录用户为官方菜谱 + 我的云端菜谱。
  • 如果当前有搜索或标签筛选，“全选”只选择当前筛选结果，避免误导出不可见项。
  • 每张菜谱可以单独勾选，导出按钮显示 导出 N 个菜谱。

数据来源

导出直接使用 getAllRecipes() 返回的 UnifiedRecipe，再映射为 JSON 包格式。映射时移除 id、source、createdAt、isOfficial、isMine 等运行时字段。

文件生成

  • 客户端使用 Blob 生成 JSON 文件并触发下载。
  • 文件名建议：cooking-plan-recipes-YYYYMMDD-HHmm.json。
  • JSON 使用 2 空格缩进，便于人工审查。

5导入设计

入口

在首页增加“导入 JSON”按钮。点击后打开导入对话框，用户选择文件后先进入预检，不立即写入。

导入预检

  1. 读取文件，限制文件类型为 .json，并限制大小，第一版上限 2MB；云端 API 在 request.json() 前检查 Content-Length，超过 2MB 直接返回 413。
  2. 解析 JSON，解析失败显示“JSON 格式不合法”。
  3. 使用共享 schema 校验结构和业务规则。
  4. 展示预检摘要：菜谱数量、标签集合、无效项数量、同名冲突数量。
  5. 只有全部通过校验后，才展示确认导入按钮。

导入归属

用户状态导入位置说明
匿名用户本地 IndexedDB使用现有 saveRecipe() 本地路径，导入后属于本地菜谱。
普通登录用户我的云端菜谱使用当前账号创建菜谱，不能导入为官方菜谱。
管理员官方菜谱当前系统没有菜谱级官方标记，管理员账号创建的云端菜谱都会被展示为官方菜谱，因此管理员导入固定为官方导入。

同名冲突策略

固定策略：不覆盖任何已有菜谱。导入时如果目标范围内已有同名菜谱，自动改名为 菜名_2、菜名_3，依次递增直到没有冲突。

原因：当前 recipes.title 没有唯一约束，官方菜谱也可能出现多个同名项。覆盖同名菜谱无法可靠判断要覆盖哪一条，第一版不提供覆盖能力。

批量写入

  • 本地导入：逐条调用本地保存逻辑，单条失败时停止并展示已导入数量。
  • 云端个人导入：调用批量导入 API，后端使用新增批量导入 service，在同一个 db.transaction() 内写入，全部成功或全部失败。
  • 管理员官方导入：调用管理员专用批量导入 API，后端校验管理员权限，并复用同一个事务内批量导入 service。

图片引用提示

  • imageUrl 为空：正常导入，无需提示。
  • imageUrl 为外部 URL：预检提示“导入后图片依赖外部地址可访问”。
  • imageUrl 为 /uploads/...：预检提示“JSON 只包含路径，不会迁移图片文件；跨环境导入后需要单独同步 uploads 文件”。
  • app 存在且不等于 cooking-plan：预检提示“该文件来源不是 cooking-plan，请确认后导入”。

6校验策略

导入校验需要放在共享纯模块中，客户端预检和服务端写入必须复用同一套规则，避免前后端不一致。

建议新增共享模块

src/lib/recipe-package.ts
  MAX_RECIPES_PER_IMPORT = 100
  MAX_INGREDIENTS_PER_RECIPE = 100
  MAX_STEPS_PER_RECIPE = 100
  exportRecipePackage(recipes): RecipeExportPackage
  parseRecipePackage(json): RecipeImportPreview
  validateRecipePackage(input): RecipeExportPackage
  normalizeImportRecipe(recipe): RecipeInput
  getImportImageWarnings(recipePackage): ImportImageWarning[]

校验层级

层级校验内容处理方式
JSON 解析文件是否为合法 JSON失败后不进入预览。
包结构version、recipes 是否存在且类型正确失败后提示文件不是有效菜谱包。
菜谱结构标题、食材、步骤、标签字段逐条收集错误，展示第几条菜谱失败。
容量规则最多 100 道菜；每道最多 100 个食材、100 个步骤、8 个标签客户端预检和服务端 API 都必须执行，防止绕过 UI 提交大 JSON。
业务规则至少 1 个食材、至少 1 个步骤、标签限制复用 recipeSchema 和 normalizeRecipeTags()。
权限规则是否允许导入到目标位置服务端最终判断，客户端只做展示。

错误展示

  • 显示总错误数，并列出前 20 条错误，避免超大文件刷屏。
  • 错误格式建议：第 3 个菜谱「水煮肉片」：至少需要一个步骤。
  • 如果标签非法，直接展示现有标签错误文案。

7接口与模块

前端组件

模块职责
RecipeBulkActions首页批量选择工具栏，负责进入/退出批量模式、全选和导出。
RecipeImportDialog文件选择、预检结果、目标选择、冲突策略、确认导入。
recipe-package.ts纯共享模块，负责导出映射、JSON schema 校验、导入规范化。

建议 API

接口权限用途
POST /api/recipes/import 登录用户 导入为当前用户的“我的菜谱”。服务端按当前用户 userId 写入。
POST /api/admin/recipes/import 管理员 导入为官方菜谱。服务端按管理员账号 userId 写入，官方身份继续由 authorRole === 'admin' 推导。

请求体

{
  "recipePackage": { "version": 1, "recipes": [] },
  "conflictStrategy": "rename"
}

conflictStrategy 第一版只支持 rename。缺省时按 rename 处理；如果调用方传入其他值，服务端返回参数错误，避免误以为支持覆盖或其他策略。

响应体

{
  "importedCount": 13,
  "renamedCount": 2,
  "recipes": [
    { "title": "紫菜炒饭_2", "originalTitle": "紫菜炒饭" }
  ]
}

匿名本地导入不需要 API，直接走 IndexedDB 保存路径。

服务端批量导入 helper

云端 API 不能循环调用现有 createRecipe()，因为 createRecipe() 自己开启事务，无法保证整个批次原子性。需要新增批量导入 service/helper，在一个事务内完成所有写入。

src/lib/actions/recipe-import.ts
  importRecipesInTx(tx, {
    userId,
    recipePackage,
    conflictStrategy: "rename"
  })

src/lib/actions/recipe-tags.ts
  syncRecipeTagsInTx(tx, recipeId, tags)
  • importRecipesInTx() 负责生成不冲突标题、插入菜谱、食材和步骤。
  • syncRecipeTagsInTx() 从当前 src/lib/actions/recipes.ts 的事务内标签同步逻辑抽出，供创建、更新和批量导入复用。
  • 批量导入 API 外层只开启一次 db.transaction()，事务内不再调用会自行开启事务的 createRecipe()。

8本地到线上迁移流程

  1. 本地以管理员身份登录。
  2. 首页筛选或选择全部官方菜谱，进入批量导出，下载 JSON。
  3. 登录线上应用的管理员账号。
  4. 点击导入 JSON。管理员导入固定写入官方菜谱。
  5. 预检通过后确认导入。线上官方菜谱为空时默认直接导入；若未来已有同名菜谱，新导入菜谱会自动使用 _2 后缀改名。
  6. 导入完成后，线上首页应能看到官方菜谱和标签筛选。

如果未来官方菜谱包含上传图片，迁移时还需要同步 public/uploads 中的图片文件，或扩展 JSON 包支持图片资产。

10风险与待确认

主要风险

  • 数据覆盖 覆盖同名菜谱无法可靠定位目标，第一版禁止覆盖，只做自动改名。
  • 图片引用 JSON 只带 imageUrl，跨环境后图片路径可能不可访问，导入预检必须提示风险。
  • 大文件 超大 JSON 可能导致浏览器卡顿或服务端慢事务，第一版限制文件大小、菜谱数量、食材数量和步骤数量。
  • 事务一致性 云端批量导入必须使用事务，避免导入一半失败。

建议默认决策

问题建议原因
同名冲突固定自动追加 _2 后缀，不覆盖当前没有标题唯一约束，覆盖风险不可控。
管理员官方导入固定导入官方菜谱当前系统通过管理员作者角色推导官方身份，无法表达管理员个人云端菜谱。
图片第一版只导出 URL，导入时展示风险提示实现简单，且避免用户误以为图片文件已迁移。
容量限制2MB、最多 100 道、每道最多 100 食材/100 步骤客户端体验和服务端事务都需要明确上限。
导出字段不导出账号和数据库 ID确保 JSON 是内容包，不是环境备份。

需要评审确认

  • 容量上限是否采用当前建议值：2MB、100 道菜、每道 100 个食材、100 个步骤。
  • 第一版是否接受图片只保留 URL，不打包图片文件，但导入时显示外部 URL 和 /uploads/... 风险提示。
