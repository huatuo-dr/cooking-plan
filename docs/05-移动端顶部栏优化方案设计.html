移动端顶部栏优化方案设计

将手机端 Header 从桌面按钮平铺改为“品牌 + 主操作 + 更多菜单”
Feature: Mobile Header Version: Draft 1 Date: 2026-07-23
目录： 背景与问题 目标范围 方案取舍 推荐布局 菜单状态 实施拆解 验收测试 风险与决策

1背景与问题

当前首页顶部栏完全沿用桌面横排布局，在手机浏览器中会把标题、登录/注册、Cooking、新建菜谱、导入 JSON、批量导出全部挤在一行，导致文字断行、按钮变形和 Header 高度异常。

当前代码触发点

  • 首页 Header 位于 src/app/page.tsx。
  • 外层使用 flex items-center justify-between，操作区使用 flex items-center gap-3。
  • 当前没有移动端专用布局，没有 sm: / md: 断点，也没有折叠菜单。
  • 按钮都是文本按钮，在窄屏下被压缩成多行甚至竖排。

截图暴露的问题

问题 表现 影响
标题被挤压 Cooking Plan 被迫换行，emoji 与文字争抢空间。 品牌识别变弱，顶部栏看起来不稳定。
操作过多 手机顶部同时显示多个文字按钮。 没有主次层级，用户难以判断最高频入口。
按钮断行 “新建菜谱”“导入 JSON”等按钮发生断行。 触控面积和视觉尺寸不可控。
Header 过高 内容撑高到接近一个大块区域。 首屏被导航占用过多，菜谱内容下移。

2目标范围

移动端核心目标
顶部栏稳定在约 56px，高频操作清晰，不再出现文字竖排或按钮挤压。
桌面端目标
保持当前完整按钮栏，不改变桌面用户已有操作路径。
交互目标
移动端只露出“新建菜谱”主操作，其余操作进入更多菜单。
实现目标
优先用 Tailwind 响应式类和少量 React 状态完成，不引入新 UI 框架。

不纳入本次范围

  • 不重做全站导航体系。
  • 不新增底部导航，除非后续单独设计。
  • 不改变菜谱列表卡片、筛选器、Cooking 页面详情布局。
  • 不改变现有 API、导入导出和批量导出业务逻辑。

3方案取舍

方案 内容 优点 问题 结论
A. flex-wrap 换行 允许现有按钮自动换到第二行或第三行。 改动最小。 Header 仍会变高，按钮排列不可预测，截图中的混乱只会变成多行混乱。 不推荐
B. 横向滚动导航 保留所有按钮，手机端横向滑动。 所有入口仍在顶部。 隐藏在右侧的按钮不易发现；顶部导航变成可滚动区域后误触成本高。 不推荐
C. 品牌 + 主操作 + 更多菜单 手机端只显示品牌、主按钮和菜单按钮，低频入口收进菜单。 稳定、清晰、触控友好，能控制 Header 高度。 需要实现移动端菜单状态和菜单项权限分支。 推荐

4推荐布局

移动端 Header

移动端采用单行、固定节奏布局：

[Cooking Plan]                         [+] [Menu]
  • 左侧品牌：优先显示 Cooking Plan，极窄屏可显示 Cooking。
  • 右侧主操作：Plus 图标按钮，跳转 /recipes/new。
  • 右侧更多菜单：Menu 图标按钮，展开菜单承载低频入口。
  • Header 高度目标约 56px，避免撑开首屏。

5菜单状态

未登录状态

位置 展示 行为
Header 主按钮 Plus 进入新建菜谱页。本地访客模式仍可创建本地菜谱。
菜单项 Cooking、导入 JSON、批量导出、登录、注册 点击后执行当前已有行为，点击菜单项后关闭菜单。

已登录状态

位置 展示 行为
Header 主按钮 Plus 进入新建菜谱页，云端保存。
菜单项 Cooking、导入 JSON、批量导出、个人中心 个人中心替代登录/注册。
本阶段不新增“管理员后台”快捷入口。管理员相关能力仍从个人中心进入，避免移动端菜单和桌面端 Header 入口不一致。

6实施拆解

  1. 拆分 Header 结构。 在 src/app/page.tsx 中将现有 Header 操作区拆成桌面完整栏和移动简化栏。桌面栏使用 hidden md:flex，移动栏使用 flex md:hidden。
  2. 移动端品牌处理。 去掉或缩小 emoji。品牌文字使用 text-xl 或更紧凑尺寸，并加 truncate / min-w-0，避免挤压右侧按钮。
  3. 主按钮使用图标。 使用 Plus 图标作为移动端新建按钮，保持 40x40 左右触控面积，设置 aria-label="新建菜谱"。
  4. 增加更多菜单状态。 使用 useState 管理 mobileMenuOpen。点击 Menu 图标打开/关闭菜单；点击菜单项后关闭菜单。
  5. 迁移低频操作。 将 Cooking、导入 JSON、批量导出、登录/注册或个人中心放入移动菜单。导入 JSON 和批量导出复用当前回调，不改变业务逻辑。
  6. 处理登录分支。 未登录显示登录/注册菜单项；已登录显示个人中心。管理员不额外增加顶部菜单项，保持“管理员能力从个人中心进入”的现有范围。
  7. 菜单关闭规则。 点击菜单外部关闭，按 Escape 关闭，点击任一菜单项后关闭。第一阶段可以不做复杂 focus trap，但基础关闭行为必须实现。
  8. 可访问性语义。 菜单按钮设置 aria-label、aria-controls 和 aria-expanded；新建等图标按钮必须有 aria-label；跳转入口使用 Link，导入 JSON、批量导出等执行动作使用 button。
  9. 保持桌面端不变。 桌面端保留现有完整按钮，不改变当前用户熟悉的入口和视觉样式。

7验收测试

桌面端

  • 宽度大于等于 768px 时，完整按钮栏仍显示。
  • 新建菜谱、Cooking、导入 JSON、批量导出、登录/注册或个人中心行为不变。

移动端

  • 320px、360px、375px、390px 宽度下 Header 高度保持约 56px，不出现两行按钮或竖排文字。
  • 左侧品牌不遮挡右侧按钮；必要时品牌文本截断。
  • 右侧只显示新建按钮和菜单按钮。
  • 点击新建按钮进入 /recipes/new。
  • 点击菜单按钮显示 Cooking、导入 JSON、批量导出和账号相关入口。
  • 点击导入 JSON 能打开现有导入弹窗。
  • 点击批量导出能进入现有批量导出模式。
  • 点击任一菜单项、点击菜单外部、按 Escape 后菜单关闭。
  • 导航菜单项使用链接语义，动作菜单项使用按钮语义；图标按钮均有可读标签。

8风险与决策

风险 处理方式
菜单项太多 只放当前 Header 已有入口，不扩展新功能。管理员能力仍通过个人中心进入，不在移动端单独新增管理员后台入口。
批量导出入口变深 批量导出是低频操作，移动端放入菜单符合优先级；桌面端不变。
标题仍然过长 使用 truncate，极窄屏可显示 Cooking 或降低字号。
菜单遮挡内容 菜单定位在 Header 右下方，宽度控制在 220px 左右，层级高于内容区域；同时要求外部点击、Escape 和菜单项点击都能关闭菜单。
可访问性不足 菜单按钮设置 aria-label、aria-controls 和 aria-expanded；图标按钮提供明确标签；导航项用 Link，动作项用 button。
推荐决策： 第一阶段只优化首页移动端 Header，不引入底部导航。等首页顶部栏稳定后，再评估 Cooking 是否需要作为全站底部导航入口。
