🍳 Cooking-Plan 初建方案

为「真实厨房场景」设计的多菜协同菜谱应用
v0.3 已定稿 可开工
文档：docs/00-初建方案文档.html · 创建 2026-06-30 · 更新 2026-06-30（v0.3）
核心洞察：传统菜谱都是一道道独立、串行的。但真实做菜时，多道菜是同时进行的—— 切菜、腌制、炒制的步骤会交织在一起。本应用要解决的就是：把多道菜的步骤按阶段合并， 再让用户手动拖拽排成贴合真实操作的"作战时间线"。

目录： ① 产品目标 ② 关键决策 ③ 技术栈 ④ 数据模型 ⑤ 页面结构 ⑥ Cooking 模式 ⑦ 实施计划 ⑧ 评审结论

📝 v0.3 变更记录（定稿）

  • 准备 + 制作合并为「统一时间线」：Cooking 视图不再把准备/制作切成两个固定分区，而是放进同一条可拖拽列表，靠颜色圆点 + 「准备/制作」小标签区分。这样"先煮饭→再切菜→回来翻炒"等跨阶段交错顺序可完整表达，贴合真实厨房。食材仍为独立勾选清单。
  • nit：cooking_session_items.recipe_id 改为可空 + ON DELETE SET NULL——删菜谱后快照计划仍完整，仅"重建"按钮失效。
  • nit：图片压缩/缩略图用服务端 sharp 处理。

📝 v0.2 变更记录（已纳入评审建议）

  • Cooking 计划改为快照：创建计划时复制当时的步骤/食材，与后续菜谱编辑解耦，消除多态外键悬空问题；详情页提供"用最新版重建"按钮。
  • 食材阶段改为勾选清单：不再可拖拽排序，改为"按菜分组的勾选清单"，v1 自然落地买菜清单（不做不可靠的数量自动求和）。
  • 合并步骤表：prep_steps + cook_steps 合并为一张 steps(phase)。
  • 颜色入模型：颜色分配到 cooking_session_items.color（会话内属性）。
  • 实施顺序调整：核心功能（菜谱 + Cooking）优先，登录鉴权下沉到核心跑通之后。
  • 补充：updated_at、ON DELETE CASCADE、zod 校验、上传校验+压缩、弱网乐观更新、核心逻辑单测。

1产品目标

功能一：制作菜谱

每道菜按 食材 / 准备 / 制作 三个部分设计，每部分可由用户自由增删步骤；每道菜可上传一张图片作为菜谱展图。

  • 食材：结构化记录——名称（必填）+ 用量（可选），如「番茄 2个」「盐 适量」
  • 准备：不开火的预处理（洗、切、腌、调酱汁）
  • 制作：开火烹饪（炒、煮、蒸、炖）

功能二：Cooking（多菜协同）

从自己的菜谱中勾选几道菜，系统按 食材 / 准备 / 制作 三个阶段把所有菜的步骤整合到一起， 用户可手动拖拽调整各步骤顺序，排成同时做多道菜时真实的操作顺序。

4数据模型

users
  id, email, password_hash, created_at, updated_at

recipes
  id, user_id → users, title, image_url, created_at, updated_at

ingredients                         食材（结构化）
  id, recipe_id → recipes ⊗CASCADE,
  name(必填), amount(可选), sort

steps                               ★ 准备/制作合并为一张表
  id, recipe_id → recipes ⊗CASCADE,
  phase('prep' | 'cook'), text, sort

cooking_sessions                    一次 Cooking 计划
  id, user_id → users ⊗CASCADE,
  name, created_at, updated_at

cooking_session_items               选了哪些菜 + 会话内分配的颜色
  id, session_id → cooking_sessions ⊗CASCADE,
  recipe_id → recipes (可空, ⊗SET NULL), color
  ↑ 删菜谱后快照仍完整，仅"重建"失效

cooking_session_steps               ★ 快照：创建时复制的准备/制作步骤
  id, session_id → cooking_sessions ⊗CASCADE,
  phase('prep' | 'cook'),
  source_recipe_title,   ← 快照：来源菜名
  color,                 ← 快照：颜色
  text,                  ← 快照：步骤文字
  sort, done

cooking_session_ingredients         ★ 快照：食材清单（按菜分组勾选）
  id, session_id → cooking_sessions ⊗CASCADE,
  source_recipe_title, color,
  name, amount, done

★ = v0.2 评审后变更。⊗CASCADE 表示父行删除时级联删除。 关系概览：用户 → 多菜谱；菜谱 → 多条食材/步骤(phase 区分准备/制作)； Cooking 会话创建时把当时的步骤与食材复制成快照，之后编辑菜谱不影响已建计划（多态外键已移除）。

6Cooking 模式交互

勾选多道菜 → 生成快照（食材清单 + 步骤） → 在统一时间线上拖拽排真实顺序 → 做菜时打勾完成

Cooking 视图分两个区：① 食材勾选清单（按菜分组，备料用）；② 准备 + 制作统一时间线（一条可拖拽列表，准备/制作步骤可自由交错）。

统一时间线示意（准备 ⇄ 制作可交错）

番茄炒蛋 制作 先把电饭煲煮上饭 ☐
青椒肉丝 准备 肉丝用淀粉腌 10 分钟 ☐
番茄炒蛋 准备 番茄切块、鸡蛋打散 ☐
青椒肉丝 制作 热油下肉丝快炒 ☐

每个步骤前用专属颜色圆点（区分菜）+ 菜名标签 + 准备/制作小标签三重标识，拖乱顺序后仍能一眼分辨来源与类型； 每步可打勾完成；调整后的顺序自动保存到该 Cooking 计划，下次可接着用。

两个区的差异（v0.3 定稿）

  • 食材：勾选清单 按菜分组列出，不可拖拽（食材是清单不是动作）；同名食材尽力归组并列展示，不强行求和。这一区即"买菜/备料清单"。
  • 准备 + 制作：统一可拖拽时间线 二者放进同一列表自由交错排序，靠小标签区分类型——表达"先煮饭→再切菜→回锅翻炒"的真实操作流。

弱网友好：打勾与拖拽采用乐观更新 + 后台同步，厨房网络差时也不卡（不做完整离线，v1 够用）。
快照与更新：计划是创建时的快照；详情页提供「用菜谱最新版重建此计划」按钮，需要时一键同步最新内容。
